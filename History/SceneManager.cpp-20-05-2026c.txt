TEXTURE RELOAD ON DISK-CACHE RESTORE (GLB + GLTF paths)

Root cause:
  After a disk-cache restore, COM textureSRVs cannot be deserialised — models[] holds
  empty SRVs. The verification block called SetupModelForRendering → LoadFallbackTexture,
  which wrote bricks.png fallback SRVs back to models[]. On the next same-session reload,
  CopyFrom copied the non-empty fallback SRVs, so the check never re-fired. Wrong texture
  was locked in permanently.

Fix — ParseGLBScene mini-parse block (~line 348):
  1. Before the mini-parse: scan scene_models[0..instanceIndex] for any loaded geometry
     model with empty textureSRVs; set texturesNeedReload flag.
  2. Widen condition from (!bAnimationsLoaded) to (!bAnimationsLoaded || texturesNeedReload).
  3. Inside the try block, after ParseMaterialsFromGLTF + ParseAnimationsFromGLTF:
     - Loop over scene_models[i] with empty textureSRVs.
     - Match models by materials[0] name against miniDoc["materials"][mi]["name"].
     - Clear all existing SRVs (textureSRVs, normalMapSRVs, metallicMapSRV,
       roughnessMapSRV, aoMapSRV) before binding.
     - Call BindGLTFMaterialTexturesToModel(mi, ...) to load real GPU textures.
     - Write restored textures/SRVs back to matching models[] entry (matched by
       cachedInstanceIndex + sourceSceneFile) so future same-session reloads copy
       real data rather than fallback placeholders.

Fix — ParseGLTFScene mini-parse block (~line 1389):
  Identical pattern applied to the GLTF (plain JSON) path.

Guards: all texture reload code is wrapped in #if defined(__USE_DIRECTX_11__).
