  SECTION 2: WHAT NEEDS FIXING (BUGS & CORRECTNESS ISSUES)
================================================================================

  [BUG-01] CRITICAL: kGlobalLightCBMinBytes magic number duplicated
  ------------------------------------------------------------------
  Files:  DX11Renderer.cpp:133  AND  DXRenderFrame.cpp:745
  Issue:  The value 1728 (minimum global light CB size required by shader) is
          hard-coded in TWO separate places as a local static const. If the
          shader layout changes and the minimum size changes, both locations
          must be updated in sync or you get DEVICE_DRAW_CONSTANT_BUFFER_TOO_SMALL
          and zero-light rendering (grey ship) again.
  Fix:    Move to a single constexpr in a shared header:
              // In e.g. ConstantBuffer.h:
              constexpr UINT kGlobalLightCBMinBytes = 1728;

  [BUG-02] CRITICAL: Sleep-based software frame cap has poor precision
  --------------------------------------------------------------------
  File:   DXRenderFrame.cpp:606-616
  Issue:  When VSync is disabled, the frame limiter calls
          std::this_thread::sleep_for(targetFrame - frameTime) to target 16ms
          (60fps). On Windows, the default system timer resolution is ~15.6ms.
          If frame rendering takes 14ms, sleep(2ms) may sleep 15.6ms — yielding
          an effective cap of ~35fps, not 60fps. This also causes frame-rate
          instability visible as micro-stutters.
  Fix:    At engine startup (main.cpp), call timeBeginPeriod(1) to set 1ms timer
          resolution. Call timeEndPeriod(1) at shutdown.
          Requires: #include <timeapi.h> and link Winmm.lib.
          NOTE: Already needed if XMMODPlayer uses winmm internally.

  [BUG-03] MODERATE: ESC key handler blocks keyboard hook thread
  -------------------------------------------------------------
  File:   KBHandlersCode.cpp:73-79 and 88-95
  Issue:  The SetKeyUpHandler lambda busy-waits on fxManager.IsFadeActive()
          with 5ms sleeps. This runs inside the WH_KEYBOARD_LL hook callback.
          Windows requires low-level hooks to return within ~300ms or the hook
          is removed and input stops working. A 1-second fade blocks the hook
          thread for its full duration.
  Fix:    Set an atomic flag (e.g., bEscapeToIntroRequested or bEscapeToQuit)
          in the key handler and process the fade+scene-switch in the main
          game loop. The hook returns immediately.

  [BUG-04] MINOR: Initialize() comment inverted (maintenance hazard)
  ------------------------------------------------------------------
  File:   DX11Renderer.cpp:188
  Issue:  The else branch (bIsResizing == false = initial startup) has the
          comment "We are resizing the window, so restart the loading sequence"
          but the else branch is the NOT-resizing (first-time init) path.
          The comment belongs on the if-branch. Wrong comments cause future
          developers to invert logic when modifying this code.
  Fix:    Swap the comments: if-branch → "Re-initialising after resize, loader
          already running"; else-branch → "Initial startup — begin asset loading".

  [BUG-05] MINOR: wrapW and wrapH are assigned but never used
  -----------------------------------------------------------
  File:   DX11Renderer.cpp:461, 470
          int wrapW = iTileSizeX - destW1;   // line 461, inside Blit2DWrapped
          int wrapH = iTileSizeY - destH1;   // line 470
  Issue:  Both variables are computed but referenced nowhere. MSVC will emit
          C4189 warnings (variable initialized but not referenced).
  Fix:    Remove both lines, or add [[maybe_unused]] if future code needs them.

  [BUG-06] MINOR: Misleading switch indentation in KBHandlersCode.cpp
  -------------------------------------------------------------------
  File:   KBHandlersCode.cpp:103-118
  Issue:  After the KEY_ESCAPE case's inner switch closes at line 103, the
          comment reads "// End of switch (keyCode)" which is wrong — it closes
          the inner scene-type switch, not the outer keyCode switch. Cases
          KEY_F2, KEY_F9, KEY_F12, etc. are syntactically correct but visually
          appear to be inside KEY_ESCAPE due to indentation level.
          Fix the comment on line 103 to read: "// End of inner switch (stSceneType)"
          and de-indent KEY_F2 and following cases to the correct outer level.

  [BUG-07] MINOR: SyncVersionFiles.ps1 is untracked in git
  ---------------------------------------------------------
  File:   cmake/SyncVersionFiles.ps1
  Issue:  New PowerShell script that duplicates cmake/IncrementVersion.cmake
          functionality but is not tracked in git. Build collaborators won't
          have it. Risk of the two scripts diverging in version format.
  Fix:    git add cmake/SyncVersionFiles.ps1 and commit it. Consider whether
          to consolidate with IncrementVersion.cmake or keep both (CMake for
          CMake builds, PS1 for VS/MSBuild builds).

  SECTION 3: CRITICAL OPTIMISATIONS
================================================================================

  [OPT-01] HIGH PRIORITY: Per-frame heap allocation in global light upload
  ------------------------------------------------------------------------
  File:   DXRenderFrame.cpp:717
  Code:   std::vector<LightStruct> globalLights = lightsManager.GetAllLights();
  Issue:  GetAllLights() returns a std::vector by VALUE every single frame.
          This allocates heap memory, copies all LightStruct data, then frees
          it — every frame, 60+ times per second. For a scene with 8 lights
          (LightStruct ~80 bytes), this is ~640 bytes allocated and freed per
          frame = ~38KB/s of heap churn, plus potential cache misses from the
          allocator touching different memory regions.
  Fix:    Option A: Change GetAllLights() to return const std::vector<LightStruct>&
          (returns reference to internal member). Zero-copy.
          Option B: Pre-declare a static/member GlobalLightBuffer glb and fill
          it directly in the render loop without the intermediate vector.
  Impact: Eliminates ~38KB/s heap churn; reduces cache pressure in render loop.

  [OPT-02] MEDIUM PRIORITY: Camera matrices recomputed inside model loop
  -----------------------------------------------------------------------
  File:   DXRenderFrame.cpp:773-775
  Code:   (inside for i = 0; i < MAX_SCENE_MODELS loop)
              scene.scene_models[i].m_modelInfo.viewMatrix       = myCamera.GetViewMatrix();
              scene.scene_models[i].m_modelInfo.projectionMatrix = myCamera.GetProjectionMatrix();
              scene.scene_models[i].m_modelInfo.cameraPosition   = myCamera.GetPosition();
  Issue:  Each getter likely copies a matrix. With MAX_SCENE_MODELS = e.g. 64,
          this is 64 * 3 function calls and matrix copies per frame. The camera
          state does not change within a frame.
  Fix:    Cache outside the loop:
              const auto viewMat  = myCamera.GetViewMatrix();
              const auto projMat  = myCamera.GetProjectionMatrix();
              const auto camPos   = myCamera.GetPosition();
          Then assign from locals inside the loop. Saves ~192 copies/frame.

  [OPT-03] MEDIUM PRIORITY: Timer resolution for VSync-off frame limiter
  -----------------------------------------------------------------------
  (See also BUG-02 above — this is both a bug and an optimisation concern.)
  File:   DXRenderFrame.cpp:606-616
  Fix:    timeBeginPeriod(1) at startup. Without it, even properly-designed
          frame limiters are capped by the 15.6ms Windows scheduler quantum.
          This also benefits std::this_thread::sleep_for calls elsewhere (e.g.
          the pause-loop in the renderer thread: DXRenderFrame.cpp:136).

  [OPT-04] LOW PRIORITY: FPS debug text rebuilt every frame
  ----------------------------------------------------------
  File:   DXRenderFrame.cpp:461-468
  Issue:  The FPS wstring is constructed from multiple std::to_wstring() calls
          on every rendered frame (not just every second). Each to_wstring()
          allocates a temporary wstring on the heap. When showDebugInfo=true
          and FPS display is active, this produces ~6 heap allocations per frame
          (FPS value, mouse x, mouse y, camera x/y/z, yaw, pitch, light count).
  Fix:    Cache the composed wstring and rebuild it only when the FPS value
          updates (the 1-second window). The camera/mouse values are fast-
          changing but tolerate 1-frame staleness in a debug overlay.
          Alternatively, use fmt::format or swprintf_s into a fixed wchar_t[].

  [OPT-05] LOW PRIORITY: LoaderTaskFinished check inside blit functions
  ----------------------------------------------------------------------
  File:   DXRenderFrame.cpp:329-337 (SCENE_GAMETITLE background)
  Issue:  threadManager.threadVars.bLoaderTaskFinished.load() is called
          multiple times per frame (once in RenderBackgroundImage, once in the
          2D block, once in the model loop). Each atomic load is a memory fence.
          Acceptable, but could cache the bool at the top of RenderFrame() for
          the frame duration.
  Fix:    const bool bLoadDone = threadManager.threadVars.bLoaderTaskFinished.load();
          Use bLoadDone throughout the frame. Trivial change, marginal gain.

  SECTION 5: CODE QUALITY OBSERVATIONS
================================================================================

  [POSITIVE]
  + Consistent use of ComPtr<> for all COM resources — correct lifetime mgmt.
  + RAII ThreadLockHelper pattern — locks always released, even on exception.
  + Extensive #if _DEBUG_ guards on debug logging — zero overhead in Release.
  + Abstract Renderer interface — engine code is renderer-agnostic.
  + SecureZeroMemory on sensitive member arrays in constructor — good security.
  + Configuration checksum with FNV-1a — tamper detection on save files.
  + exceptionHandler.RecordFunctionCall() in hot paths — crash tracing enabled.

  [CONCERNS]
  - ExceptionHandler::RecordFunctionCall() called in RenderFrame() hot path
    (DXRenderFrame.cpp:100). If this writes to a vector or similar on every
    frame, it may accumulate unbounded memory. Verify it has a ring-buffer
    or fixed-size mechanism.
  - BeginDraw() called inside a try-catch that catches std::exception only.
    D2D methods return HRESULT and do NOT throw C++ exceptions — the catch
    block will never fire for D2D failures. Only HRESULT checks are meaningful.
  - static Microsoft::WRL::ComPtr<ID3D11RasterizerState> wireframeRS declared
    inside RenderFrame() (DXRenderFrame.cpp:220) — this ComPtr is never
    populated and exists as dead code. Will generate no harm but is confusing.
  - The global bResizing (bool) and bResizeInProgress (atomic<bool>) coexist
    as extern declarations — having two separate resize flags risks getting
    them out of sync. Worth auditing which one is authoritative.

  [DOCUMENTATION]
  + ReleaseInfo.md is well-maintained with dated changelog entries.
  + Docs/ directory has per-class Example Usage .md files for all major systems.
  + Renderer.h has a detailed ASCII class relationship diagram and flow notes.
  - To-Do.txt lists planned features but is not dated — hard to track
    what was added vs. what is still pending.
